## Problem

Most multi-agent demos look impressive because the workflow is narrow, the input is controlled, and failure has no organizational cost. Enterprise systems operate in the opposite environment: ambiguous requests, changing policy, legacy data, security boundaries, and teams that need accountability.

<Callout title="Core idea">
  The hard part is not creating more agents. The hard part is designing
  responsibility, memory, evaluation, and control around them.
</Callout>

## Failure Patterns

- Agents are given overlapping responsibilities.
- Tool access is granted before policy is modeled.
- Memory is treated as a chat transcript instead of a governed product surface.
- Evaluation begins after deployment.
- Human approval is added as a UI patch instead of a workflow primitive.

## Architecture Response

<Architecture title="Production control loop">

1. Route the user goal to the right workflow.
2. Decompose work into accountable agent roles.
3. Bind every tool to policy, logging, and rollback behavior.
4. Evaluate intermediate outputs, not only final answers.
5. Escalate decisions that affect money, access, safety, or customers.

</Architecture>

## Tradeoffs

More control adds latency and platform complexity. Less control creates adoption risk. The right answer is usually not "fully autonomous" or "fully manual"; it is scoped autonomy with explicit escalation.
